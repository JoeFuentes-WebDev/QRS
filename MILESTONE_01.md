# MILESTONE_01.md

## SECTION 1 — HUMAN SUMMARY

This milestone runs the Prisma schema migration to create the V3 database structure on Neon. It defines all entities — Seller, Product, Order, OrderItem — with the correct fields for Stripe Connect Express, shipping mode toggle, and order state machine. This must come first because every subsequent milestone (checkout, webhooks, onboarding, etc.) depends on the schema being live. Estimated complexity: low-medium. No application logic, no UI. Schema definition and migration only.

Define and apply the complete V3 Prisma schema to the Neon Postgres database. This includes:

### 6. Schema Specification

Implement exactly the following in `prisma/schema.prisma`:

**Datasource:**
```
provider = "postgresql"
url      = env("DATABASE_URL")
```

**Generator:**
```
provider = "prisma-client-js"
```

**Enums:**

```prisma
enum ShippingModel {
  BAKED_IN
  SEPARATE
}

enum OrderStatus {
  PENDING
  ACCEPTED
  DECLINED
  FULFILLED
  REFUNDED
}
```

**Seller model fields:**
- `id` — String, @id, @default(cuid())
- `clerkUserId` — String, @unique
- `storeName` — String
- `slug` — String, @unique
- `notificationEmail` — String
- `stripeConnectAccountId` — String? (nullable — not set until Connect onboarding starts)
- `stripeConnectOnboarded` — Boolean, @default(false)
- `ShippingModel` — ShippingModel, @default(BAKED_IN)
- `shippoApiKey` — String? (nullable — optional, enables SEPARATE mode)
- `createdAt` — DateTime, @default(now())
- `updatedAt` — DateTime, @updatedAt
- Relations: `products Product[]`, `orders Order[]`

**Product model fields:**
- `id` — String, @id, @default(cuid())
- `sellerId` — String (FK to Seller)
- `name` — String
- `description` — String
- `price` — Int (stored in pence/cents, integer — no floating point money)
- `images` — String[] (array of Cloudinary URLs)
- `tags` — String[] (AI-generated)
- `category` — String
- `stock` — Int, @default(0)
- `published` — Boolean, @default(false)
- `createdAt` — DateTime, @default(now())
- `updatedAt` — DateTime, @updatedAt
- Relations: `seller Seller @relation(fields: [sellerId], references: [id])`, `orderItems OrderItem[]`

**Order model fields:**
- `id` — String, @id, @default(cuid())
- `sellerId` — String (FK to Seller)
- `buyerEmail` — String
- `status` — OrderStatus, @default(PENDING)
- `stripePaymentIntentId` — String? (nullable — set on checkout completion)
- `stripeSessionId` — String, @unique (used for idempotency — must be unique)
- `createdAt` — DateTime, @default(now())
- `updatedAt` — DateTime, @updatedAt
- Relations: `seller Seller @relation(fields: [sellerId], references: [id])`, `items OrderItem[]`

**OrderItem model fields:**
- `id` — String, @id, @default(cuid())
- `orderId` — String (FK to Order)
- `productId` — String (FK to Product)
- `quantity` — Int
- `priceSnapshot` — Int (price at time of purchase, in pence/cents)
- `createdAt` — DateTime, @default(now())
- Relations: `order Order @relation(fields: [orderId], references: [id])`, `product Product @relation(fields: [productId], references: [id])`

---

### 7. Layer Rules Reminder

1. **No application logic in this milestone.** Schema definition only. No service functions, no API routes, no UI components.
2. **Price fields are integers (pence/cents), never Float or Decimal.** Floating point money is forbidden. Use `Int` for `price` and `priceSnapshot`.
3. **`stripeSessionId` is `@unique` on Order.** This is the idempotency key for the Stripe webhook handler. It must be unique at the schema level.

---

### 8. Engineering Rules

**DO:**
- Use `cuid()` for all `@id` fields
- Use `@updatedAt` on every model's `updatedAt` field
- Use `String?` (nullable) for `stripeConnectAccountId` and `shippoApiKey` — these are optional at creation time
- Use `String[]` for `images` and `tags` arrays — Prisma supports this on Postgres/Neon
- Apply `@unique` to `Seller.clerkUserId`, `Seller.slug`, and `Order.stripeSessionId`
- Run `prisma generate` after schema is written
- Run `prisma migrate dev --name v3_initial` to create and apply the migration locally
- Verify migration applies cleanly against the Neon DATABASE_URL

**DO NOT:**
- Use `Float` or `Decimal` for any price field — use `Int` only
- Create any additional models not specified above (no ShippingConfig, no OrderEvent, no AnalyticsEvent)
- Add any Prisma middleware or extensions in this milestone
- Create a seed file in this milestone
- Modify any existing application files
- Add `@@index` directives — index optimization is V4
- Use `autoincrement()` for IDs — use `cuid()` only

---

### 9. Ambiguity Protocol

If any of the following conditions are true, **stop and record the ambiguity** in the Open Questions section rather than assuming:

- `DATABASE_URL` is not present in `.env.local` or environment — do not proceed with migration
- There is an existing `prisma/schema.prisma` that contains models not listed in this milestone spec — record what was found, do not delete silently
- `prisma migrate dev` fails due to a connection error — record the error, do not attempt to work around it
- Postgres array type (`String[]`) is rejected by the Neon instance — record and stop

No safe assumption exists for database connectivity failures. Record and stop.
